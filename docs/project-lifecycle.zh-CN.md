# 项目生命周期中文说明

这份文档描述的是:

一个 Claude 协作任务, 从“接到需求”到“收尾提交”, 推荐按什么生命周期推进。

## 推荐总流程

1. 读全局规则
2. 判断任务类型
3. 判断是否需要规划
4. 判断是否需要 skill
5. 判断是否需要 workflow
6. 缩小上下文
7. 执行本轮任务
8. 验证结果
9. 更新文档和记忆
10. 提交、推送、PR

## 1. 读全局规则

先读:
- `CLAUDE.md`
- 当前相关中文说明

目的不是全读仓库, 而是先知道:
- 默认怎么协作
- 什么是高风险
- 这次有没有权限直接做

## 2. 判断任务类型

先判断当前任务属于哪一类:
- 扩模块
- 加功能
- 修 bug
- 事故排查
- 文档整理
- 任务拆分

这个判断会影响你该先读 skill 还是 workflow。

## 3. 判断是否需要规划

如果任务小到一句话能说清 diff, 直接做。

如果任务有这些特征, 建议先规划:
- 范围不清楚
- 风险高
- 跨多个模块
- 需要多 agent

## 4. 判断是否需要 skill

如果这件事属于“项目里会反复遇到的一类问题”, 优先看 skill。

例如:
- AssetBundle 崩溃排查
- 战斗模块扩写
- 模块任务拆分

## 5. 判断是否需要 workflow

如果你已经知道做哪类事情, 但希望推进顺序稳定, 优先看 workflow。

例如:
- 扩模块
- 修 bug
- 处理资源事故

## 6. 缩小上下文

只加载这轮最需要的内容:
- 全局规则
- 当前模块文档
- 当前模块 `CLAUDE.md`
- 当前 task
- 当前 skill 或 workflow
- 当前模块相关代码

不要把无关模块一起带进来。

## 7. 执行本轮任务

原则:
- 一次只做一轮任务
- 保持 scope 清楚
- 尽量保留公共接口稳定
- 小步验证

## 8. 验证结果

至少回答这些问题:
- 本轮目标是否完成
- 哪些接口保持稳定
- 哪些内容故意没做
- 有没有残余风险

## 9. 更新文档和记忆

如果这轮改动改变了长期规则, 更新:
- `.claude/memory/decisions.md`

如果这轮暴露了反复出现的坑, 更新:
- `.claude/memory/known-issues.md`

如果模块阶段变化了, 更新:
- 模块 `CLAUDE.md`
- `docs/systems/<module>.md`

## 10. 提交、推送、PR

提交前:
- 检查 scope
- 检查 docs 是否同步
- 检查高风险改动是否有说明

推送前:
- 确认验证已做
- 确认风险已记录

PR 时:
- 说明目标
- 说明改动范围
- 说明验证
- 说明残余风险
