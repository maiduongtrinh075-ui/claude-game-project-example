# 常见陷阱与性能优化

这份文档总结的是:

在使用 Claude 处理大项目时, 最容易出现的几类问题, 以及如何避免它们。

## 1. 上下文溢出

### 症状

- Claude 忘记早期指令
- 重复犯同样的错误
- 响应越来越慢
- 注意力开始分散

### 为什么会这样

不是因为模型突然变差, 而是因为当前会话承载了太多信息。

当任务、文件、历史讨论、日志、规划都堆在一个长会话里时, 重点会被稀释。

### 预防方式

- 定期开始新会话
- 使用子 agent 隔离不同任务
- 只读当前任务真正相关的文件
- 优先看规则、文档、task, 再读实现代码

### 推荐做法

如果你已经从:
- 战斗扩写
- AB 崩溃排查
- GitHub 配置
- 文档结构设计

一路聊到一起了, 通常就该开新会话。

## 2. 模糊指令

### 症状

- Claude 反复问澄清问题
- 做出来的东西和预期不一致
- 需要多轮纠正

### 为什么会这样

因为任务描述太泛了, Claude 只能自己猜。

例如:
- “帮我完善一下”
- “优化这个模块”
- “看看哪里有问题”

这类说法都缺少边界。

### 预防方式

- 提供具体上下文
- 指明允许修改的范围
- 给出验收标准
- 引用现有 skill、workflow 或 task

### 更好的说法

不要说:

```text
完善战斗系统
```

更推荐说:

```text
阅读 CLAUDE.md、docs/systems/combat.md、src/gameplay/combat/CLAUDE.md、tasks/in-progress/combat-phase-1.md。
只扩写 combat 模块, 增加技能选择和敌人自动行动。
不要改存档结构, 不要改全局事件名。
```

## 3. 过度规划

### 症状

- 简单任务花了太长时间
- 规划比实现还长
- Claude 一直停留在“准备阶段”

### 为什么会这样

不是所有任务都值得先做复杂计划。

有些任务其实一句话就能说清楚 diff, 这时继续长篇规划只会拖慢节奏。

### 预防方式

- 小任务直接做
- 只在不确定或高风险时规划
- 如果一句话能描述改动, 就可以跳过重计划

### 判断方式

例如这些通常不需要长规划:
- 改一个字段名
- 补一个 README
- 加一个日志
- 修一个明显的 null 检查

## 性能优化

### 1. 减少上下文使用

核心原则:

不要把“项目里存在的所有信息”都塞进当前任务。

推荐做法:
- 只加载当前模块
- 优先读高密度信息源
- 需要时再补充远处依赖

### 2. 减少无效输出

长命令输出、海量日志、无关文件内容, 都会拖慢判断。

推荐做法:
- 只看摘要
- 只看关键片段
- 只定位和当前问题相关的栈或文件

### 3. 保持会话干净

当一个话题已经结束, 就不要继续把它背在新任务上。

推荐做法:
- 大话题切新会话
- 不同问题分开处理
- 复杂任务拆成 task 或子 agent

### 4. 使用子 agent 并行处理

适合:
- 可并行拆开的任务
- 各自负责不同模块
- 需要主 agent 收敛结果

不适合:
- 小任务
- 强耦合修改
- 多个人会改同一批文件

## 一句话总结

Claude 不是“喂得越多越好”, 而是:

越聚焦, 越稳定。

所以最好的策略通常是:
- 任务具体
- 上下文精简
- 简单任务直接做
- 大任务再拆 skill、workflow、task 和 multi-agent
