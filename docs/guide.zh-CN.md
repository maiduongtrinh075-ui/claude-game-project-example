# 中文说明

这个仓库是一个完整示例, 专门演示大项目里如何同时使用三层协作结构:

- 全局 `CLAUDE.md`
- 模块 `CLAUDE.md`
- 单轮任务文件

它最适合这种情况:
- 项目很大, 一次写不完
- 某个功能需要从 0 到 1 起步
- 后面还要让 Claude 多轮继续扩写
- 你不希望 Claude 顺手把别的系统一起改乱

## 三层结构怎么分工

### 1. 全局 `CLAUDE.md`

它负责全项目规则:
- 这个项目是什么
- 默认先读哪些文件
- 哪些区域是稳定区
- 哪些区域允许扩写
- 默认不要做什么

它解决的是“Claude 进入项目时怎么快速对齐全局背景”。

### 2. 模块 `CLAUDE.md`

它负责某一个大功能的局部规则。

比如战斗模块自己的 `CLAUDE.md` 可以写:
- 当前阶段做到哪
- 这个模块下一步按什么顺序扩
- 哪些接口不能动
- 哪些修改属于本轮允许范围

它解决的是“Claude 进入某个长期施工中的功能区时, 怎么不重新乱猜”。

### 3. 单轮任务文件

它负责这一轮到底做什么。

例如:
- 增加 3 个技能
- 增加敌人自动行动
- 不改存档
- 只允许修改战斗和 battle UI

它解决的是“本轮只吃一口, 不把范围越做越大”。

## 推荐工作流

1. 先选一个模块
2. 读全局 `CLAUDE.md`
3. 读模块自己的 `CLAUDE.md`
4. 读这一轮任务文件
5. 只按任务改代码
6. 如果长期规则变了, 写进 `.claude/memory/decisions.md`

## 什么时候需要模块 `CLAUDE.md`

不是每个文件夹都需要。

适合新建模块 `CLAUDE.md` 的情况:
- 这个功能要做很多轮
- 这个功能本身复杂
- 这个功能从 0 到 1 还没完成
- 它有很明确的扩写顺序
- 它有不能乱动的接口

不必新建模块 `CLAUDE.md` 的情况:
- 临时小功能
- 一两次就做完
- 用一个 `README.md` 就能说清楚

## 你以后怎么给 Claude 下指令

推荐这种写法:

```text
阅读 CLAUDE.md、docs/systems/combat.md、src/gameplay/combat/CLAUDE.md、tasks/in-progress/combat-phase-1.md。
只扩写 combat 模块, 实现任务文件里的目标。
不要改存档结构, 不要改全局事件名, 只允许修改战斗模块和相关 battle UI。
```

这比“完善战斗系统”稳定很多。

## 多 Agent 怎么理解

多 Agent 不是默认必开, 它适合“任务能明确拆开”的情况。

适合多 Agent 的情况:
- 一个 agent 查 bug, 一个 agent 补日志
- 一个 agent 改 gameplay, 一个 agent 改 UI
- 一个 agent 写文档, 一个 agent 写测试

不适合多 Agent 的情况:
- 任务太小
- 模块边界不清楚
- 多个人会改到同一批文件
- 需求本身还没讲清楚

推荐规则:
- 每个 agent 只负责一个清晰区域
- 不要让多个 agent 同时改同一个模块
- 由一个主 agent 负责最终集成和验收

你可以把它理解成:

- 单 Agent: 适合大多数日常开发
- 多 Agent: 适合可并行拆分的大任务
